# Java Train Ticketing System


## Project Architecture

The application runs entirely in memory without relying on an external database, utilizing standard Java Collections. The codebase is organized into three main layers:

* **`com.luca.model` (Entities):** Contains the core data structures (`Train`, `Station`, `Route`).
* **`com.luca.data` (Repository):** Manages the central `DataStore` and the `DataInitializer` which seeds the application with a predefined railway network.
* **`com.luca.service` (Business Logic):** Contains the services that enforce the application's rules (`BookingTickets`, `RouteFinder`, `Administrator`).

---

## Features & Assignment Fulfillment

### A) Ticket Booking System (`BookingTickets.java`)
* Allows users to book one or multiple tickets for a specific train.
* **Overbooking Prevention:** Continuously tracks train capacity and available seats. If a booking request exceeds the available capacity, it is automatically rejected.
* **Email Simulation:** Upon successful booking, a simulated confirmation email is generated and appended to a local log file (`emails_log_confirmation.txt`).

### B) Route Finder (`RouteFinder.java`)
* Given a departure and arrival station, the system searches the predefined routes.
* Supports finding **Direct Routes**.
* Supports finding routes requiring a **Changeover** (1 stop) by identifying intersecting stations between different routes.
* Outputs a clear error message if there is **no possible link** between the two stations.

### C) Administrator Panel (`Administrator.java`)
* Ability to add new trains and routes to the central `DataStore`.
* Ability to generate reports showing the number of booked seats versus the total capacity for any train.
* **Delay Reporting:** Admins can flag a train as delayed. This action triggers a simulated system alert that writes a warning email to all passengers in `emails_log_delays.txt`.

## Input / Output Examples

The application logic is driven through the `MainClass.java` file. Below are examples of the exact inputs used and the corresponding outputs generated by the system.

### 1. Booking a Ticket (Success)
**Input:**
> `bookingSystem.bookTicket(1, "client1@email.com", 2);`

**Output (Console):**
> Successful booking of 2 for train 1

**Output (File - emails_log_confirmation.txt):**
> Booking Email Confirmation
> 
> To: client1@email.com
> Subject: Booking Confirmation
> Message: Your ticket/s for train 1 are confirmed. Have a safe journey!
> ----------------------

### 2. Booking a Ticket (Overbooking Prevention)
*Note: Train 4 has a maximum capacity of 2 seats.*

**Input:**
> `bookingSystem.bookTicket(4, "test@email.com", 3);`

**Output (Console):**
> Failed booking of 3 for train 4

### 3. Route Finder: Direct Route
**Input:**
> `routeFinder.findRoute(new Station("Paris"), new Station("Lyon"));`

**Output (Console):**
> Searching for a journey from Paris to Lyon...
> -> Success: Direct route found!
>    Take Train ID 1 (Route ID 1)

### 4. Route Finder: One Changeover
**Input:**
> `routeFinder.findRoute(new Station("Paris"), new Station("Monaco"));`

**Output (Console):**
> Searching for a journey from Paris to Monaco...
> -> Success: Route found with 1 changeover!
>    1. Take Train ID 1 from Paris to Marseille
>    2. CHANGEOVER at Marseille
>    3. Take Train ID 2 to Monaco

### 5. Route Finder: No Possible Link
**Input:**
> `routeFinder.findRoute(new Station("Monaco"), new Station("Manchester"));`

**Output (Console):**
> Searching for a journey from Monaco to Manchester...
> -> Error: No possible link between Monaco and Manchester

### 6. Administrator: Reporting a Delay
**Input:**
> `admin.reportDelay(1);`

**Output (Console):**
> Admin: Train 1 is delayed.
> -> SYSTEM: Sending delay warning emails to all passengers on Train 1...
> -> Delay emails saved to emails_log_delays.txt

### 7. Administrator: Checking Bookings
**Input:**
> `admin.showBookingsForTrain(1);`

**Output (Console):**
> Admin Report: Train 1 has 2 booked seats out of 150
